## Обзор архитектуры Dockerfile

На странице рассматривается архитектура `Dockerfile` и **ключевой терминологии**, необходимой для эффективной работы с образами.

### 1. Архитектура Dockerfile
`Dockerfile` является текстовым файлом с набором инструкций для сборки образа, архитектура которого основана на **слоистой модели**, который обрабатывается следующим образом:

1. Базовый образ (Base Image), является отправной точной

2. Каждая последующая инструкция создает новый **слой (Layer)**

3. Слои кэшируются - если строка не изменилась, Docker использует кэщ.

4. Copy-on-write (COW) - при запуске контейнера поверх слоев образа создается тонкий записываемый слой.

, что графически можно представить следующим образом:

```bash
+--------------------------------+   Layer 6: CMD (метаданные)
|  Записываемый слой (контейнер) |
+--------------------------------+   
|  Layer 5: RUN npm start   | <- Изменяет слой 4
+---------------------------+   
|  Layer 4: COPY . /app     | <- Добавляет файлы
+---------------------------+   
|  Layer 3: RUN apt install | <- Устанавливает пакеты
+---------------------------+   
|  Layer 2: WORKDIR /app    | <- Меняет контекст
+---------------------------+   
|  Layer 1: FROM ubuntu     | <- Базовый слой (только чтение)
+---------------------------+
```

### 2. Используемая терминология
`Image (Образ)` - Неизменяемый шаблон (read-only) с ОС, кодом, зависимостями.

`Container (Контейнер)` - Запущенный экземпляр образа с записываемым слоем.

`Layer (Слой)` - Результат выполнения одной инструкции **Dockerfile**.

`Registry` - Хранилище образов (Docker Hub, GitHub, ...)

`Tag (Тег)` - Версия образа, `myapp:1.0`, `myapp:latest`.

`Build context` - Путь к файлам, передаваемый демону при выполнении `docker build`.

### 3. Инструкции Docker

**FROM** - определяет базовый/родительский образ, является обязательной первой строкой Dockerfile `FROM python:3.11-slim`.

**RUN**	- Выполняет команду во время сборки (создает слой)	`RUN apt update && apt install -y curl`.

**COPY** - Копирует файлы из контекста в образ	`COPY . /app`.

**ADD** - Как `COPY` + умеет распаковывать `.tar` и скачивать по URL `ADD https://example.com/file.tar.gz/tmp/`.

**WORKDIR** - Устанавливает рабочую папку в нутри контенера (создает, если нет)	`WORKDIR /app`.

**ENV**	- Устанавливает переменные окружения `ENV NODE_ENV=production`.

**ARG** - Переменная, используемая только для сборки (не сохраняется в образе) `ARG VERSION=1.0`

**EXPOSE** - Документирует IP-порт без публикации на хосте `EXPOSE 8080`.

**CMD**	- Команды, выполняемые внутри образа после его запуска `CMD ["node", "server.js"]`.

**ENTRYPOINT** - Основной процесс контейнера (не переопределяется легко) `ENTRYPOINT ["python"]`.

**VOLUME** - Создает точку монтирования для данных с хоста в контейнер `VOLUME /data`.

**HEALTHCHECK**	- Проверяет, жив ли контейнер	`HEALTHCHECK CMD curl -f http://localhost/`.

*LABEL*	- определение метаданных образа (версия, maintainer) `LABEL version="1.0"`.

### 4. Типовые архитектурные паттерны Dockerfile

#### Паттерн 1: Одностадийная сборка (простая)
```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json .
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "index.js"]
```

**Достоинства:** Простота использования
**Недостатки:** Итоговый образ содержит инструменты сборки и исходные коды, что обуславливает наличие неиспользуемых в дальнейшем файлов и большой размер.

#### Паттерн 2. Многостадийная сборка (Multistage build)
Является золотым стандартом при разработке программного обеспечения.
```dockerfile
# СТАДИЯ 1: СБОРКА
FROM golang:1.21 AS builder
WORKDIR /src
COPY go.mod go.sum .
RUN go mod download
COPY . .
RUN go build -o myapp .

# СТАДИЯ 2: ФИНАЛЬНЫЙ ОБРАЗ
FROM alpine:latest
RUN apk add --no-cache ca-certificates
WORKDIR /root/
COPY --from=builder /src/myapp .
EXPOSE 8080
CMD ["./myapp"]
```

В результате выполнения сборки проекта в первом образе и последующем переносе только скомпилированного исполняемого файла (без среды сборки, git-файлов, исходных кодов программы) во второй образ, можем получить разницу в размерах второго (итогового) как соотношение 1,2Gb к 15 Мб.

#### Паттерн 3, разделение сборок на dev/prod версии
```dockerfile
# dev-образ (горячая перезагрузка)
FROM node:18 AS development
CMD ["npm", "run", "dev"]

# prod-образ (минимальный)
FROM node:18-alpine AS production
COPY --from=development /app/dist ./dist
RUN npm ci --only=production
CMD ["node", "dist/index.js"]
```

### 5. Лучшие практики/best practices, или что важно для архитектуры
#### 5.1. Объединение `RUN` кодманд
**Плохо**, что приведет к образованию двух слоев в образе
```dockerfile
    RUN apt-get update
    RUN apt-get install curl -y
```

**Хорошо**, в результате в образе будет один слой
```dockerfile
RUN apt-get update && apt-get install curl -y
```

#### 5.2. Использование конкретных тегов

**Плохо***, в связи с возможным появлением новой версии образа, что определяется как *нестабильность*.
```dockerfile
FROM python:latest
```

**Хорошо**, повторяемо при сборке образа в любое время.
```dockerfile
FROM python:3.11-slim
```

#### 5.3. Минимизирование числа слоев
Учитывайте, что каждая инструкция `RUN`, `COPY` или `ADD` используемая в Dockerfile приводит к образованию отдельного слоя.
Используйте агрегацию команд не в ущерб читабельности файла.

#### 5.4. Соблюдение порядка инструкций от менее к более изменяемым
Сначала `WORKDIR`, `ENV`, `RUN apt-get`, потом `COPY`-копирование кода, чтобы кэш не сбрасывался при каждом измененеии кода.

#### 5.5. Не запускайте контейнер с правами `root`
```dockerfile
RUN addgroup -g 1001 -S appgroup && adduser -u 1001 -S appuser -G appgroup
USER appuser
```
##### 5.6. Использование `.dockerignore`
Исключите `.git`, `node_modules`, `*.log`, `Dockerfile`, `.env`.

### 6. Типовые команды для работы с Docker
```bash
# Сборка образа, при навигации
# в каталоге с Dockerfile
$ docker build -t custom_image:1.0 .

# Сборка с указанием контента и файла
$ docker build -f docker/Dockerfile.prod -t custom_image:prod ./src

# Обзор образов в локальном хранилище Docker
$ docker images

# Просмотреть слои образа
$ docker history <image-name>

# Запуск с переопределением CMD
$ docker run <image-name> echo "Текстовое сообщение"

# Запуск с переопределением ENTRYPOINT
$ docker run --entrypoint /bin/sh <image-name>
```
